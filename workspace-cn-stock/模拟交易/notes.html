1. 回测引擎：逐日回放机制

回测核心是 backtest.py 中的 run_backtest()，采用逐日事件驱动（event-driven daily replay）框架。

主循环伪代码

for date in trading_days:
    snapshot = fetch_historical(date, watchlist)   # ① 获取当日快照
    portfolio.update_prices(snapshot)              # ② 刷新持仓市值
    decisions = strategy.decide(snapshot, portfolio)# ③ 策略产生决策
    for d in decisions:                            # ④ 执行买卖
        portfolio.buy(...) / portfolio.sell(...)
    portfolio.net_value_history.append(...)         # ⑤ 记录净值

绩效指标计算

指标实现方式
最大回撤遍历净值序列，维护峰值，(peak - current) / peak 取最大
夏普比率逐日收益率 → 年化收益 & 年化波动（×√242），减无风险利率 2%
胜率维护每只股票买入均价，卖出价 > 均价算赢

数据预取优化

回测前调用 prefetch_range(start, end, watchlist) 批量拉取整段历史并按日缓存到 data/daily_cache/，避免逐日逐股请求 AkShare。

2. 2025H1 回测复盘

绩效概览

指标值
总收益率+1.58%
最终资产¥1,015,812
最大回撤2.29%
夏普比率0.33
交易次数23 笔（买15 / 卖8）
胜率50%

核心问题：资金利用率极低

期末持仓 7 只，总市值约 ¥34 万，现金 ¥67 万，资金利用率仅 33%。

为什么持股这么少？三个限制叠加：
  1. buy_pct = 0.05 — 每笔只投总资产的 5%（约 ¥5 万），单票仓位极轻
  2. max_trades_per_day = 2（含卖出）— 4/7 那天卖了 4 只就用光额度，买入被挤占
  3. 已持有不重复买 — if code in portfolio.positions: continue，同一只股票不允许加仓

10 只观察池 × 每只最多一笔 5 万 = 上限 50 万仓位。加上止盈触发快（平安、招行、恒瑞都 15% 止盈了），卖后要等下次跌 5% 信号才买回，中间大量时间空仓等信号。

4/7 关税冲击复盘

4 月 7 日一天内触发 4 笔止损（美的 -8.3%、宁德 -15%、隆基 -10.5%、海康 -8%），但 max_trades_per_day=2 限制了当日买入。次日 4/8-4/9 迅速逢低重新建仓，最终这波先卖后买反而降低了成本。

改进方向

参数当前建议
buy_pct0.050.08 ~ 0.10
max_trades_per_day23 ~ 4
加仓逻辑禁止允许，受 max_single_position_pct: 0.20 约束
买入阈值20日跌 > 5%蓝筹可放宽到 3~4%

3. 双策略架构：Rule vs Claude

RuleStrategy 和 ClaudeStrategy 都继承 Strategy 基类，实现同一个接口：

decide(snapshot: MarketSnapshot, portfolio: Portfolio) → List[Decision]

对 backtest.py 主循环来说，不关心决策怎么产生的，调用方式完全一样。

RuleStrategy — 硬编码规则

  • 买入：近20日跌超 5% → 买入 5% 仓位
  • 卖出：浮盈 ≥ 15% 止盈 / 浮亏 ≥ 8% 止损

特点：快、确定性强、可大规模回测。

ClaudeStrategy — LLM 决策

调用链：

decide(snapshot, portfolio)
  → _build_prompt()     # 拼接：持仓 + 行情 + strategy.md 全文
  → _call_claude()      # subprocess: claude -p <prompt>
  → _parse_response()   # 解析 JSON → List[Decision]

每个交易日调一次 Claude Code CLI（print 模式），半年约 120 次调用。能理解 strategy.md 的完整策略规则，但慢且成本高。

strategy.md 完整规则 vs RuleStrategy 实现差距

策略文档描述RuleStrategy 实现？
PE < 近5年25分位未实现
股息率 > 3%未实现
站上20日均线 + 均线拐头未实现（只看了20日涨跌幅）
成交量放大 1.2 倍未实现
PE > 75分位卖出未实现
跌破20日线连续3日卖出未实现
加仓逻辑未实现（已持有直接跳过）
总仓位 60-80% 控制未实现
止盈 15% / 止损 8%已实现
近20日跌幅买入已实现
结论 strategy.md 是完整策略蓝图，RuleStrategy 只实现了最简化的骨架。ClaudeStrategy 理论上能理解全部规则，但依赖 LLM 的输出质量。

4. 选股能力缺失分析

RuleStrategy 没有选股能力。整个系统的股票范围在数据层就被锁死了：

config.json watchlist (手动写死10只蓝筹)
       ↓
engine.py fetch_historical(date, watchlist)  # 只拉这10只
       ↓
snapshot.quotes = {这10只的当日行情}
       ↓
RuleStrategy.decide()  # 只能在这10只里挑

snapshot 里有什么数据？

字段RuleStrategy 用了？
close用了（算涨跌幅 + 下单价格）
open, high, low没用
volume, amount没用
pe, pb没用
change_pct没用（自己算的）

如果要加选股

方案 A：扩大 watchlist + 定期筛选 — 把观察池扩到沪深300成分股，在 decide() 前加 screen() 步骤用 PE/PB/股息率筛选。缺点是数据量大，AkShare 逐股拉取慢。

方案 B：独立选股模块 — 新增 screener.py，定期（每月）用全市场数据筛选，输出更新 config.json 的 watchlist。回测时按月动态更换观察池。

定位 当前系统本质是「给定股票池的择时系统」，不是选股系统。

5. 20日均线扩展：需要改什么？

strategy.md 定义的卖出条件「跌破20日均线且连续3日未收回」，数据层其实已经够了。

现有数据 vs 策略需求

卖出条件需要什么现有系统
浮盈 > 15%close + 持仓成本有，已实现
浮亏 > 8%同上有，已实现
PE > 近5年75分位历史 PE 序列没有，需要外部数据
跌破20日均线3日近20日 close有（_price_history 保留30天）
结论：不需要扩展数据层 _price_history 已保留每只股票最近 30 天的 close 价格，sum(prices[-20:]) / 20 即可算出均线。只需在 strategy.py 加十几行逻辑。

需要改的两件事

① 加计算逻辑（strategy.py 卖出检查部分）：

prices = self._price_history.get(code, [])
if len(prices) >= 20:
    ma20 = sum(prices[-20:]) / 20
    # 最近3日是否都在均线下方
    below_days = sum(1 for p in prices[-3:] if p < ma20)
    if below_days >= 3:
        decisions.append(Decision(action="sell", ...))

② 解决冷启动（backtest.py）：

回测从 2025-01-01 开始时 _price_history 为空，前20个交易日无法算均线。改法：预取数据多往前取一个月。

# backtest.py — 多取30天做热身
prefetch_range("2024-12-01", end, watchlist)

PE 历史分位才是大工程

「PE > 近5年75分位数」需要拉取每只股票 5 年的 PE 历史序列。当前 AkShare 只取单日 PE，这是一个需要额外数据源的扩展。

各层职责边界

层知道什么不知道什么
engine怎么拉数据、缓存策略、持仓
strategy行情 + 当前持仓 → 产生决策怎么执行交易、费用规则
portfolio交易规则、仓位管理行情从哪来、为什么要买
dashboard怎么渲染数据策略逻辑、数据来源

层与层之间通过三个数据对象传递：MarketSnapshot → Decision → Portfolio 状态。backtest.py 是粘合剂，把四层串成循环。
